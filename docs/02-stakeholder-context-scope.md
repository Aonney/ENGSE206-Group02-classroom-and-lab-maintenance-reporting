# 1. Problem Frame (Revised)

## 1.1 สถานการณ์ปัจจุบัน

ปัจจุบันผู้ใช้งานมักแจ้งปัญหาผ่านหลายช่องทาง เช่น แจ้งอาจารย์ ส่งข้อความผ่านแชต หรือแจ้งเจ้าหน้าที่ด้วยวาจา ส่งผลให้ข้อมูล
การแจ้งปัญหากระจัดกระจายและไม่เป็นมาตรฐาน บางครั้งข้อมูลสําคัญ เช่น ตําแหน่ง อาการ หรือรูปภาพประกอบ ไม่ครบถ้วน
ทําให้เจ้าหน้าที่ต้องสอบถามข้อมูลเพิ่มเติมก่อนดําเนินการ

นอกจากนี้ ปัญหาเดียวกันอาจถูกแจ้งจากหลายคน ผู้แจ้งไม่สามารถติดตามสถานะของงานซ่อมได้ และงานบางกรณีต้องส่งต่อ
หลายหน่วยงาน ส่งผลให้การดําเนินงานล่าช้าและกระทบต่อการเรียนการสอน โดยเฉพาะกรณีเหตุขัดข้องระหว่างการเรียน

---

## 1.2 ใครได้รับผลกระทบ

- **นักศึกษา/ผู้ใช้งานห้อง:** ไม่ทราบว่างานดำเนินถึงขั้นใด แจ้งซ้ำโดยไม่รู้ว่ามีคนแจ้งแล้ว
- **อาจารย์ผู้สอน:** การเรียนสะดุดหากอุปกรณ์ใช้งานไม่ได้และแก้ไขไม่ทัน
- **เจ้าหน้าที่เทคนิค:** ต้องสอบถามข้อมูลเพิ่ม ได้รับข้อมูลไม่ครบ หรือมีการแจ้งซ้ำ
- **ผู้ดูแลอาคาร/ผู้บริหาร:** ไม่เห็นสถานะรวมของปัญหา และประเมินประสิทธิภาพการดำเนินงานได้ยาก

---

## 1.3 Problem Statement

> ผู้ใช้งานห้องเรียนและห้องปฏิบัติการยังไม่สามารถแจ้งปัญหาและติดตามงานซ่อมได้อย่างเป็นระบบ เนื่องจากการแจ้งปัญหากระจายอยู่หลายช่องทางและข้อมูลไม่เป็นมาตรฐาน ส่งผลให้เกิดการแจ้งซ้ำ การติดตาสถานะทำได้ยาก และการดำเนินงานซ่อมล่าช้า

---

### 1.4 ผลลัพธ์ที่ต้องการ (โดยยังไม่กำหนด solution)

- ผู้ใช้งานสามารถแจ้งปัญหาได้อย่างเป็นมาตรฐานและให้ข้อมูลที่จำเป็นครบถ้วน
- ผู้แจ้งสามารถติดตามความคืบหน้าของงานซ่อมได้อย่างชัดเจน
- ลดการแจ้งปัญหาซ้ำจากหลายช่องทาง
- เจ้าหน้าที่ได้รับข้อมูลที่เพียงพอในการวิเคราะห์และดำเนินการแก้ไขปัญหา
- ทำให้การประสานงานระหว่างผู้แจ้งและเจ้าหน้าที่มีประสิทธิภาพมากขึ้น
- ผู้บริหารสามารถติดตามภาพรวมของปัญหาและประเมินประสิทธิภาพการดำเนินงานได้จากข้อมูลที่เชื่อถือได้

---

## 1.5 สิ่งที่ทีมยังต้องเรียนรู้

- เกณฑ์การจัดลำดับความเร่งด่วนของงาน (Priority / SLA)
- ใครเป็นผู้รับผิดชอบการปิดงาน
- วิธีจัดการกรณีแจ้งปัญหาซ้ำ
- ช่องทางการแจ้งเตือนที่ผู้ใช้ต้องการ
- ข้อมูลขั้นต่ำที่เจ้าหน้าที่ต้องใช้ก่อนเริ่มซ่อม

## 2.1 Stakeholder Profiles

#### Stakeholder: นักศึกษา / ผู้ขอใช้

- **Goal / need:** แจ้งปัญหาได้สะดวกและรวดเร็ว ติดตามสถานะการซ่อมได้ และทราบเมื่อปัญหาได้รับการแก้ไข
- **Pain point:** ไม่ทราบว่ามีการรับเรื่องแล้วหรือไม่ ไม่สามารถติดตามความคืบหน้าของงานได้ และอาจแจ้งปัญหาซ้ำโดยไม่ทราบว่ามีผู้แจ้งก่อนแล้ว
- **Concern / risk:** ข้อมูลส่วนบุคคลรั่วไหล สิทธิ์ไม่เหมาะสม หรือการเชื่อมระบบภายนอกทำไม่ได้
- **Information this stakeholder knows:** อาการของปัญหา สถานที่เกิดเหตุ เวลา และหลักฐาน เช่น รูปภาพ
- **Influence / decision power:** edium — เป็นผู้แจ้งปัญหาหลัก แต่ไม่มีอำนาจตัดสินใจด้านการดำเนินงาน
- **Open questions for Week 3:** ผู้ใช้ต้องการติดตามสถานะในรูปแบบใด? ต้องการได้รับการแจ้งเตือนเมื่อใด? ข้อมูลใดที่ผู้ใช้คิดว่าจำเป็นในการแจ้งปัญหา?

#### Stakeholder: อาจารย์ผู้สอน

- **Goal / need:** ให้อุปกรณ์พร้อมใช้งานก่อนและระหว่างการเรียนการสอน และลดผลกระทบจากเหตุขัดข้องในชั้นเรียน
- **Pain point:** ไม่ทราบความคืบหน้าของการแก้ไขปัญหา และหากอุปกรณ์เสียระหว่างสอน อาจทำให้การเรียนสะดุด
- **Concern / risk:** การอนุมัติผิดทรัพยากร การลืมติดตามคืน หรือผู้ใช้ไม่ทราบเงื่อนไข
- **Information this stakeholder knows:** ผลกระทบต่อการเรียน ความเร่งด่วนของเหตุการณ์ และช่วงเวลาที่เกิดปัญหา
- **Influence / decision power:** Medium — สามารถแจ้งเหตุและให้ข้อมูล แต่ไม่มีอำนาจจัดการงานซ่อม
- **Open questions for Week 3:** ทรัพยากรชนิดใดต้องอนุมัติ? เกณฑ์ใดใช้ตัดสินใจ? ต้องบันทึกการส่งมอบ/คืนระดับใด?

#### Stakeholder: เจ้าหน้าที่เทคนิค

- **Goal / need:** ได้รับข้อมูลการแจ้งที่ครบถ้วน สามารถจัดลำดับงานและดำเนินการซ่อมได้อย่างมีประสิทธิภาพ
- **Pain point:** ข้อมูลการแจ้งไม่ครบ ต้องสอบถามเพิ่มเติม และพบการแจ้งปัญหาซ้ำ
- **Concern / risk:** การวิเคราะห์ปัญหาผิดพลาด งานล่าช้า หรือส่งต่องานผิดหน่วยงาน
- **Information this stakeholder knows:** ขั้นตอนการซ่อม ประเภทของปัญหา ระยะเวลาการดำเนินงาน และข้อมูลที่จำเป็นต่อการซ่อม
- **Influence / decision power:** High — เป็นผู้รับผิดชอบการดำเนินงานและอัปเดตสถานะของงาน
- **Open questions for Week 3:** ข้อมูลขั้นต่ำที่ต้องใช้ก่อนเริ่มซ่อมคืออะไร? เกณฑ์การจัดลำดับความสำคัญของงานเป็นอย่างไร? ใครเป็นผู้มีสิทธิ์ปิดงาน?

#### Stakeholder: ผู้ดูแลอาคาร/ผู้บริหาร 

- **Goal / need:** เห็นภาพรวมของปัญหาและประสิทธิภาพการดำเนินงาน เพื่อนำไปใช้ในการวางแผนและตัดสินใจ
- **Pain point:** ไม่สามารถติดตามสถานะงานซ่อมทั้งหมดได้ และประเมินผลการดำเนินงานได้ยาก
- **Concern / risk:** การตอบสนองล่าช้า ส่งผลต่อคุณภาพการเรียนการสอน และขาดข้อมูลสำหรับการวางแผน
- **Information this stakeholder knows:** นโยบายการบริหาร งบประมาณ ภาพรวมของงานซ่อม และตัวชี้วัดการดำเนินงาน
- **Influence / decision power:** High — กำหนดนโยบาย อนุมัติแนวทาง และประเมินผลการดำเนินงาน
- **Open questions for Week 3:** ต้องการรายงานหรือสถิติรูปแบบใด? ตัวชี้วัด (KPI) ที่ใช้ประเมินงานซ่อมคืออะไร? ต้องการข้อมูลสรุปบ่อยเพียงใด?

# 4. Scope Statement

## In Scope

- การแจ้งปัญหาอุปกรณ์ภายในห้องเรียนและห้องปฏิบัติการ
- การบันทึกรายละเอียดปัญหา เช่น ประเภทอุปกรณ์ หมายเลขห้อง และรูปภาพประกอบ
- การรับเรื่องและจัดการรายการแจ้งซ่อมของเจ้าหน้าที่เทคนิค
- การติดตามและอัปเดตสถานะของงานซ่อม
- การส่งต่องานไปยังหน่วยงานที่เกี่ยวข้อง (กรณีจำเป็น)
- การแจ้งเตือนผู้แจ้งเมื่อสถานะของงานเปลี่ยนแปลง
- การแสดงรายงานและสถิติภาพรวมสำหรับผู้ดูแลอาคารหรือผู้บริหาร

---

## Out of Scope

- ระบบบริหารสินทรัพย์ทั้งหมด
- ระบบจัดซื้อจัดจ้าง
- ระบบงบประมาณ
- ระบบควบคุมอาคารอัตโนมัติ
- การติดตามตำแหน่งด้วย GPS

---

## Constraints

- ใช้งานเฉพาะห้องเรียนและห้องปฏิบัติการ
- ไม่จัดเก็บข้อมูลส่วนบุคคลเกินความจำเป็น
- ต้องปกปิดข้อมูลส่วนบุคคลที่อาจปรากฏในภาพแนบ
- ระบบต้องสามารถรองรับการส่งต่องานระหว่างหน่วยงาน
- โครงการพัฒนาภายในระยะเวลา 1 ภาคการศึกษา

---

## Assumptions

- ผู้ใช้งานสามารถแนบรูปภาพได้
- เจ้าหน้าที่สามารถอัปเดตสถานะได้อย่างสมํ่าเสมอ
- ผู้แจ้งต้องการรับการแจ้งเตือนเมื่อสถานะเปลี่ยน
- ห้องเรียนทุกห้องมีรหัสหรือหมายเลขที่ใช้ระบุตําแหน่งได้
- การแจ้งซํ้าสามารถตรวจสอบได้จากข้อมูลที่ผู้ใช้กรอก
- หน่วยงานที่เกี่ยวข้องสามารถรับงานที่ส่งต่อจากเจ้าหน้าที่เทคนิคได้

# 5. Privacy, Ethics, Security and Responsible AI

| Topic | Decision |
|---|---|
| **Data Minimization** | เก็บเฉพาะข้อมูลที่จำเป็นต่อการแจ้งซ่อม เช่น ชื่อผู้แจ้ง รหัสนักศึกษา/รหัสบุคลากร อีเมล หมายเลขห้อง รายละเอียดปัญหา และรูปภาพประกอบ |
| **Access Control** | นักศึกษาและอาจารย์เห็นเฉพาะรายการแจ้งของตนเอง เจ้าหน้าที่เทคนิคเห็นงานที่รับผิดชอบ ผู้ดูแลอาคารหรือผู้บริหารเข้าถึงข้อมูลสรุปและรายงานตามสิทธิ์ |
| **Privacy** | ปกป้องข้อมูลส่วนบุคคล และหลีกเลี่ยงการแสดงข้อมูลที่ไม่จำเป็น โดยเฉพาะข้อมูลที่อาจปรากฏในภาพแนบ เช่น ใบหน้าหรือข้อมูลส่วนบุคคล |
| **Security** | ผู้ใช้ต้องยืนยันตัวตนก่อนเข้าใช้งาน ระบบกำหนดสิทธิ์ตามบทบาท และจัดเก็บข้อมูลอย่างปลอดภัย รวมถึงเข้ารหัสรหัสผ่าน |
| **Ethics** | ใช้ข้อมูลเพื่อวัตถุประสงค์ในการแจ้งซ่อม ติดตามงาน และปรับปรุงการให้บริการเท่านั้น ไม่ใช้ข้อมูลเพื่อการเฝ้าติดตามหรือวัตถุประสงค์อื่นที่ไม่เกี่ยวข้อง |
| **Responsible AI** | ใช้ AI เพื่อช่วยจัดทำเอกสาร วิเคราะห์ความต้องการ และตรวจสอบภาษา โดยทีมเป็นผู้ตรวจสอบความถูกต้องทั้งหมด และไม่ใช้ AI สร้างข้อมูลหรือข้อสรุปที่ไม่มีหลักฐานจาก Stakeholders |

# 6. Tabletop Studio Feedback Action

## Feedback Received

- ควรแยกบทบาทและความรับผิดชอบของผู้มีส่วนได้ส่วนเสียแต่ละกลุ่มให้ชัดเจน
- ควรกำหนดขอบเขตของระบบ (System Boundary) ให้ชัดว่าอะไรอยู่ในและนอกระบบ
- ควรแยกข้อเท็จจริง (Facts) ออกจากข้อสมมติ (Assumptions)
- ควรเพิ่มคำถาม (Open Questions) เพื่อใช้ในการสัมภาษณ์ Stakeholders ใน Week 03

## What the Team Changed

1. ปรับปรุง Stakeholder Profiles ให้ระบุ Goal, Pain Point, Influence และ Open Questions ของแต่ละกลุ่ม
2. ปรับ System Context และ System Boundary ให้สอดคล้องกับขอบเขตของโครงการ
3. ทบทวน In Scope และ Out of Scope ให้ชัดเจนขึ้น
4. เพิ่ม Open Questions เพื่อใช้เก็บข้อมูลจากผู้มีส่วนได้ส่วนเสียในสัปดาห์ถัดไป
